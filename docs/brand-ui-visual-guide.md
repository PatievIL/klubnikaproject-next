# KlubnikaProject Brand UI Visual Guide

Этот документ нужен как опорная visual-инструкция для сайта `klubnikaproject`. Он не заменяет тексты страниц и не описывает маркетинговую стратегию. Его задача: зафиксировать визуальную систему, чтобы любые следующие сессии, дизайнеры или верстальщики работали в одной логике.

## 1. Scope

Документ покрывает:

- общий визуальный характер бренда;
- правила работы с логотипом;
- цветовую систему;
- типографику;
- фирменные формы;
- фотостиль;
- композицию;
- UI-систему;
- адаптацию `desktop -> tablet -> mobile`;
- визуальную логику ключевых страниц сайта.

Документ не покрывает:

- посты и соцсети;
- презентации;
- упаковку;
- мерч;
- препресс;
- Pantone;
- офлайн-носители.

## 2. Core Visual Thesis

`KlubnikaProject` должен выглядеть не как “агро-маркетинг”, а как собранная рабочая система для клубничной фермы в controlled environment.

Базовая формула визуала:

- натуральность без деревенской стилистики;
- технологичность без неона и sci-fi;
- дороговизна через порядок, воздух и точность;
- продукт и инженерная логика важнее декоративности;
- интерфейс должен выглядеть уверенно, спокойно и чисто.

## 3. Brand Character

Визуальные ассоциации, которые нужно удерживать:

- свежесть;
- чистота;
- порядок;
- реальный продукт;
- производственная точность;
- зрелость;
- спокойная уверенность.

Не должно быть ощущения:

- инфобизнеса;
- “фермерской ярмарки”;
- псевдо-люкса;
- техно-неона;
- шаблонного SaaS-интерфейса;
- перегруженной лендинговой упаковки.

## 4. Non-Negotiables

То, что нельзя ломать без отдельного решения:

- шрифт `Exo 2` как основная гарнитура;
- темно-зеленый как системный цвет бренда;
- светлый теплый фон как базовая поверхность;
- бежевый как теплый акцент;
- скругленные формы без агрессивной геометрии;
- реальные фотографии продукта, фермы, рук, узлов, стеллажей;
- текстовая доминанта слева и визуальная справа в широких секциях;
- чистая иерархия и умеренный ритм.

## 5. Existing Asset Sources

Использовать только существующие бренд-ассеты из проекта:

- `assets/brand-lockup-green.svg`
- `assets/brand-lockup-beige.svg`
- `assets/brand-berry-green.svg`
- `assets/brand-berry-beige.svg`
- `assets/favicon.svg`
- `assets/app-icon.svg`

### Current Primary Digital Logo

Текущая принятая основная версия логотипа для сайта — это не статичный `brand-lockup-*`, а рабочий digital lockup:

- знак ягоды `brand-berry-*`;
- двухстрочное название `Klubnika / Project`;
- плотная верхняя регистра с `Exo 2`;
- спокойный descriptor ниже.

Эта версия уже используется в шапке и на ключевых first screens и считается основной для текущего сайта.

### Asset Priority

- в основном desktop/mobile header использовать текущий site lockup;
- в компактных UI-зонах использовать `brand-berry-*`;
- `brand-lockup-*` считать резервной или вспомогательной версией, а не обязательной по умолчанию;
- альтернативные logo-направления хранить отдельно в `documents/logo/`.

### Prohibited

- не менять пропорции знака ягоды;
- не применять эффекты, glow и тени к логотипу;
- не использовать случайные новые шрифтовые пары;
- не вводить новую логосистему в прод без отдельного решения.

## 6. Color System

Базовые токены уже заданы в `styles.css` и их нужно сохранять как источник истины.

### Primary Tokens

- `--green: #004d2b`
- `--green-2: #0f653e`
- `--accent: #ffd5a0`
- `--bg: #f5ecdf`
- `--page-bg: #f5ecdf`
- `--surface: #fbf4e8`
- `--surface-2: #efe1cc`
- `--surface-3: #fffaf2`
- `--text: #153224`
- `--muted: #586359`
- `--line: rgba(0, 77, 43, 0.14)`

### Color Roles

- `green`:
  основной брендовый цвет, CTA, крупные акценты, лого на светлом фоне.
- `accent`:
  теплый акцент, поддержка логотипа на темном фоне, вторичные маркеры.
- `page-bg`:
  главная базовая среда, не белая и не серая, а теплая и мягкая.
- `surface` / `surface-2` / `surface-3`:
  уровни карточек, форм, секционных контейнеров, модулей.
- `text`:
  основной текст, всегда глубокий и спокойный, не чисто черный.
- `muted`:
  вторичный текст, подписи, вспомогательные пояснения.

### Usage Rules

- Не вводить холодные акцентные цвета без отдельной задачи.
- Не использовать черный как основной цвет интерфейса.
- Не строить крупные блоки на чисто белом фоне, если это не нужно для контраста.
- Не смешивать несколько ярких акцентов в одном экране.
- Один экран = один главный цветовой акцент.

## 7. Dark Theme Rules

В `styles.css` уже есть темная тема. Ее нужно считать вторичной, а не основной.

Принципы:

- dark mode не должен менять характер бренда;
- фон должен оставаться спокойным и глубоким, а не “OLED-black”;
- бежевый в dark mode не должен превращаться в грязно-желтый;
- контраст должен оставаться высоким, но не резким;
- карточки и поверхности разделяются за счет теплых темных слоев, а не тяжелых теней.

## 8. Typography

Основная гарнитура: `Exo 2`, fallback: `Arial, sans-serif`.

### Typeface Behavior

- заголовки плотные, крупные, с минимальным шумом;
- основной текст короткими абзацами;
- длинные текстовые полотна не использовать;
- буквы не растягивать трекингом без причины;
- избегать декоративных курсивов и контрастных вторичных гарнитур.

### Current Type Logic

- `h1`: крупный, плотный, `font-weight: 800`, `letter-spacing: -0.045em`;
- `h2`: чуть свободнее, но также плотный, `font-weight: 800`;
- `h3`: утилитарный, для модулей и карточек;
- `lead`: крупный лид-абзац только в hero и ключевых блоках;
- `sublead`: вторичный supporting copy.

### Text Rules

- Основной текст почти всегда выравнивать влево.
- Не делать широкие строки без ограничения ширины.
- Не использовать центрирование для длинных абзацев.
- Не делать из текста декоративный фон.
- Если абзац можно сократить на 30%, его нужно сократить.

## 9. Shape Language

Визуальная система держится на мягкой прямоугольной геометрии и форме ягоды.

### Base Geometry

- крупные контейнеры: мягкие скругления;
- карточки: уверенная округлость, без “bubble UI”;
- кнопки: компактные, собранные, с прямым действием;
- плашки: округлые, но не игрушечные.

### Radius Tokens

Использовать текущие токены:

- `--radius-xl: 36px`
- `--radius-lg: 26px`
- `--radius-md: 18px`

Не ставить:

- острые техничные углы в основном UI;
- сверхкруглые формы “для дружелюбности”;
- микс из нескольких конфликтующих радиусов в одном блоке.

## 10. Shadow and Depth

Тень в проекте должна быть почти незаметной и служебной.

Текущий базовый токен:

- `--shadow: 0 14px 30px rgba(0, 11, 48, 0.08)`

Принципы:

- тень помогает отделить поверхность;
- тень не должна быть главным визуальным эффектом;
- глубина строится в первую очередь на цветовых слоях, воздухе и границах;
- не использовать модные тяжелые glassmorphism-тени.

## 11. Logo Rules

### Priority

- Основной для сайта считается текущий digital lockup:
  знак ягоды + двухстрочное название `Klubnika Project`.
- Компактный знак использовать только там, где полный lockup становится слишком мелким.
- `brand-lockup-*` — допустимая вспомогательная версия, но не источник истины для шапки сайта.

### Background

- На светлом фоне: зеленый логотип.
- На темном или зеленом фоне: бежевый логотип.
- На фото: только в спокойной зоне или на подложке.

### Prohibited

- не растягивать;
- не обводить;
- не добавлять glow;
- не делать gradient fill;
- не класть на шумный фон без разделения;
- не использовать как watermark по всему экрану.

### Current Website Rule

Если задача касается именно сайта `klubnikaproject`, приоритет такой:

1. текущий site lockup;
2. compact berry sign;
3. запасные logo-исследования из `documents/logo/`;
4. `brand-lockup-*` только как вспомогательный исторический ассет.

## 12. Photography Direction

Фото — это не украшение. Это доказательство продукта, процесса и среды.

### What to Show

- реальная ягода;
- стеллажи и проходы;
- ферма в controlled environment;
- руки, сбор, обслуживание;
- полив, свет, конструктив;
- чистые производственные сцены;
- крупные планы фактуры ягоды и листа.

### What to Avoid

- случайные lifestyle-фото;
- улыбающиеся стоковые команды без отношения к продукту;
- дачная эстетика;
- грядки, если страница не про outdoor;
- пересатурированные HDR-изображения;
- грязные серо-синие фото.

### Photo Treatment

- температура: нейтральная или слегка теплая;
- красный должен быть сочным, но не неоновым;
- зеленый должен быть глубоким и натуральным;
- пересветы и выжженные блики лучше убирать;
- если кадр шумный, лучше использовать его как вторичный, а не главный.

## 13. Composition System

### Main Layout Rule

На широких экранах:

- слева смысл;
- справа визуал;
- ниже понятный следующий шаг.

### Information Hierarchy

Каждый экран должен отвечать на три вопроса:

1. Что это?
2. Почему это важно?
3. Что делать дальше?

### Section Rhythm

Экран не должен быть монотонным. Нужен ритм:

- плотный hero;
- дальше более спокойный блок;
- затем секция с карточками;
- затем доказательный или маршрутный блок;
- затем CTA.

### Layout Rules

- один экран = один главный акцент;
- не складывать в hero три равных оффера;
- не распылять внимание на 5 карточек без ведущей;
- любой сложный блок сначала собирается смыслом, потом визуалом.

## 14. Grid and Spacing

Использовать текущую логику контейнера:

- `--container: 1220px`
- основная ширина секций через `.container`
- внешняя ширина: `min(var(--container), calc(100% - 32px))`

### Spacing Logic

- крупные секции: `84px` по вертикали;
- tighter-секции: `60px`;
- внутренний паддинг framed-блоков: `34px`;
- сеточный gap: `24px`.

### Practical Rule

Если блок тесный, сначала увеличивать отступ, а не уменьшать шрифт.

## 15. UI Principles

UI должен ощущаться как часть бренда, а не как чужой шаблон.

Главные признаки правильного UI:

- чистый фон;
- ясная иерархия;
- немного акцентов;
- короткие CTA;
- ровный ритм карточек;
- спокойные бордеры;
- минимум декоративных эффектов.

## 16. Buttons

В проекте уже есть базовая кнопочная система, ее нужно продолжать.

### Button Roles

- `.btn-primary`: главное действие;
- `.btn-secondary`: вторичное действие;
- `.btn-ghost`: текстовое действие без визуального веса.

### Button Rules

- на экран не ставить больше двух заметных CTA рядом;
- primary всегда один;
- secondary не должен спорить с primary;
- подписи короткие и предметные;
- не использовать общие слова вроде `Узнать подробнее`, если можно написать точное действие.

### Visual Rules

- кнопки не должны быть слишком круглыми;
- hover легкий, без эффектного шоу;
- текст внутри не должен быть капсом без причины;
- кнопки должны выглядеть инженерно, а не “luxury premium”.

## 17. Cards

Карточка в `KlubnikaProject` должна быть модулем решения, а не плиткой из маркетплейса.

### Card Anatomy

- понятный заголовок;
- короткое описание;
- один outcome или следующий шаг;
- один action;
- при необходимости изображение или kicker.

### Card Rules

- не делать слишком высокий декоративный хедер;
- не смешивать несколько смысловых уровней;
- не перегружать карточку иконками;
- не писать длинные абзацы внутри карточек.

## 18. Forms

Форма не должна выглядеть “технически случайной”.

### Form Principles

- спокойный фон;
- четкие поля ввода;
- понятные подписи;
- один маршрут действия;
- короткие пояснения рядом с полями;
- достаточная высота кликабельных элементов.

### Avoid

- лишние рамки;
- визуальный шум;
- 8 обязательных полей без объяснения;
- слишком яркие ошибки и состояния без иерархии.

## 19. Icons and Micrographics

Иконки должны быть простыми, ясными и близкими к геометрии фирменных элементов.

Принципы:

- один стиль на весь экран;
- без сложной детализации;
- без объемных 3D-иконок;
- без техно-линий и неоновых штрихов;
- иконка должна поддерживать смысл, а не украшать пустоту.

## 20. Navigation

Навигация должна выглядеть взрослой и собранной.

### Current Direction

- логотип слева;
- навигация и CTA справа;
- выпадающие меню структурные, а не декоративные;
- первый экран не должен уезжать в overly-app-like UI.

### Navigation Rules

- верхний бар не должен быть перегружен;
- в меню должен читаться порядок, а не список ссылок;
- dropdown должен выглядеть как продуманный модуль, а не случайная белая коробка;
- навигация должна поддерживать маршрут пользователя, а не просто перечислять разделы.

## 21. Hero Rules

Hero — главный визуальный узел сайта. Он не должен быть одновременно витриной, каталогом и FAQ.

### Hero Must Include

- один сильный `H1`;
- короткий лид;
- не более двух CTA;
- один главный визуал;
- короткую навигацию к следующему шагу.

### Hero Must Not Include

- длинные полотна текста;
- 4+ CTA;
- 5 равных карточек;
- тяжелые декоративные слои;
- конкурирующие визуалы.

### Visual Balance

- текстовый блок должен быть первым;
- визуал справа должен усиливать, а не дублировать текст;
- если визуал сложный, текст нужно еще сильнее упростить.

## 22. Section Templates

Ниже шаблоны секций, которые можно повторять по проекту.

### 22.1 Split Hero

- слева: eyebrow, H1, lead, CTA;
- справа: one dominant visual + one secondary support visual;
- ниже: 2–3 route cards или outcomes.

### 22.2 Proof Grid

- короткий заголовок;
- supporting copy;
- grid 3x или 4x;
- карточки с четким различием.

### 22.3 Process Section

- заголовок;
- 3–5 шагов;
- каждый шаг: номер, суть, результат;
- без лишних иллюстраций.

### 22.4 CTA Block

- короткий оффер;
- 1 основной CTA;
- 1 вторичный CTA максимум;
- контрастная плашка или framed container.

## 23. Page-Specific Visual Direction

### Home

Главная должна быть маршрутизатором.

Визуально:

- один сильный hero;
- дальше route-логика;
- затем блоки доверия и сценариев;
- затем товарные или сервисные направления;
- затем ясный CTA.

Не превращать главную:

- в каталог всего;
- в блог;
- в тяжелую корпоративную презентацию.

### Farm

Страница расчета фермы должна выглядеть как инженерный вход.

Визуально:

- меньше декоративных фото;
- больше схем, модулей, этапов;
- карточки с логикой выбора;
- строгая и спокойная подача.

### Shop

Магазин должен быть чище и более утилитарным, чем главная.

Визуально:

- сильнее роль фильтров и карточек;
- меньше имиджевости;
- больше ясности по категориям;
- карточка товара не должна быть похожа на промо-баннер.

### Seeds

Страница посадочного материала должна выглядеть как инструмент выбора, а не садовый журнал.

Визуально:

- реальный продукт;
- структурная информация;
- аккуратные сравнительные блоки;
- минимум дачной романтики.

### Consultations / Study

Эти страницы ближе к экспертной услуге.

Визуально:

- больше воздуха;
- меньше визуального шума;
- выше роль текста и аргумента;
- фото как доказательная поддержка, а не центр композиции.

### Calc

Калькулятор — это утилитарный инструмент.

Визуально:

- максимальная ясность;
- четкие формы;
- хороший контраст;
- таблицы и вводимые поля должны быть важнее декора.

## 24. Responsive Adaptation

### Desktop

- split layouts;
- hero с визуалом справа;
- grid 3–4 колонки;
- nav c dropdown;
- больше воздуха между смысловыми блоками.

### Tablet

- часть 2-колоночных секций остается split;
- сложные grids переходят в 2 колонки;
- hero визуал уменьшается по весу;
- вторичные карточки могут уходить ниже.

### Mobile

- почти все секции строятся в одну колонку;
- сначала смысл, затем визуал;
- hero-статистику и вторичные карточки упрощать;
- кнопки растягивать на удобную ширину;
- сложные декоративные сетки упрощать до 1 главного блока.

### Mobile-First Priority Order

Если экран не помещается, убирать по порядку:

1. вторичный декоративный элемент;
2. дублирующий supporting visual;
3. лишний подзаголовок;
4. вторичную карточку;
5. только потом пересобирать типографику.

Нельзя:

- просто уменьшить все равномерно;
- сохранять desktop-grid в miniature;
- ставить слишком мелкий lockup;
- оставлять CTA ниже первого критического скролла без причины.

## 25. What Future Sessions Should Preserve

Если следующая сессия делает редизайн, доработку страницы или новый блок, она должна сохранить:

- фирменные цвета и их роли;
- типографику `Exo 2`;
- спокойный премиальный, но не пафосный характер;
- теплую светлую базу;
- зеленый как основную систему действий;
- мягкую геометрию и текущие радиусы;
- реальные фотографии controlled-environment фермы;
- split-layout логику для широких экранов;
- адаптацию с упрощением, а не с механическим масштабированием.

## 27. Fast Visual QA Checklist

Перед любой публикацией или переделкой проверять:

- Экран выглядит как часть одного бренда?
- Есть один главный акцент?
- Логотип читается и не спорит с фоном?
- Цвета не вышли за фирменную систему?
- Фото реальные и релевантные?
- UI не перегружен лишними эффектами?
- CTA понятны и не дублируют друг друга?
- Мобильная версия не разваливает иерархию?
- Экран больше помогает выбрать, чем впечатляет?
