# KlubnikaProject Calc Prompt

Рабочий стандарт для страницы `/calc`.

## Роль страницы

`/calc` — это не checkout и не lead-gate. Это инструмент первичной квалификации проекта.

Страница должна помогать человеку:

1. понять примерный состав фермы;
2. увидеть рамку по смете и экономике;
3. понять, когда расчёта уже достаточно, а когда нужен человек;
4. перейти в следующий шаг без ощущения, что форма ушла в пустоту.

## Базовая логика

- сначала объясняем ценность;
- потом показываем, кому это подходит;
- потом ведём по короткому `wizard`;
- контакты спрашиваем в конце;
- после завершения показываем понятный результат и следующий шаг.

## Что страница не должна делать

- не смешивать расчёт с доставкой, оплатой и checkout;
- не ставить длинную форму контактов в первом экране;
- не продавать “доступ к калькулятору”;
- не обещать точный бизнес-план без исходных данных;
- не писать инфобизнесовый текст.

## Обязательная структура

1. `Hero`
2. `Что получите`
3. `Кому подходит`
4. `Как работает расчёт`
5. `Wizard`
6. `Предпросмотр результата`
7. `Когда нужен не калькулятор, а подбор`
8. `Доверие`
9. `FAQ`
10. `Финальный CTA`

## UX-правила

- `wizard` вместо длинной анкеты;
- progress bar обязателен;
- каждый шаг объясняет, зачем задаётся вопрос;
- можно вернуться назад без потери данных;
- mobile должен сохранять тот же сценарий, а не превращаться в длинную простыню;
- после завершения должен быть виден следующий понятный шаг.

## Тон

- прямой;
- спокойный;
- практичный;
- без пафоса;
- без AI-шоу;
- дорогой по смыслу, а не по эпитетам.

## Формулировки, которые подходят

- `Поймёте, из чего складывается проект.`
- `Это предварительная рамка, а не финальная проектная смета.`
- `Чем точнее вводные, тем полезнее результат.`
- `Если объект сложный, калькулятор — это первый шаг, а не весь ответ.`
