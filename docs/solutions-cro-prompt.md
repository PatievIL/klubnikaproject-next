# KlubnikaProject Ready Solutions Prompt

Рабочий стандарт для страниц `готовых решений / комплектов`.

## Роль этих страниц

Это не обычные SKU-страницы. Это промежуточный слой между магазином, калькулятором и консультацией.

Пользователь должен быстро понять:

1. что это за решение;
2. для какого сценария оно подходит;
3. что входит;
4. что не входит;
5. что ещё потребуется;
6. можно ли купить это сразу или сначала нужен подбор.

## Главная логика

Готовое решение не должно маскироваться под обычную розницу.

Разделяем:

- малый типовой комплект → `Купить`;
- проектный комплект → `Получить состав и смету`;
- решение под объект → `Обсудить запуск`;
- сомневающийся пользователь → `Нужен подбор`.

## Что нельзя делать

- не прятать состав комплекта;
- не скрывать, что не входит;
- не ставить `Добавить в корзину` на сложное проектное решение;
- не обещать окупаемость, урожайность и финальный результат без подтверждения;
- не писать “идеально для всех”.

## HUB-страница должна включать

1. `Hero`
2. сценарии использования готовых решений
3. линейку решений
4. блок `Как выбрать комплект`
5. сравнение решений
6. честный блок `Что обычно забывают`
7. связку с калькулятором и подбором
8. `FAQ`
9. финальный CTA

## DETAIL-страница должна включать

1. верх страницы с типом решения и CTA
2. `Кому подходит`
3. `Что решает`
4. `Что входит`
5. `Что не входит`
6. `Что ещё обычно докупают`
7. `Под какие условия подходит`
8. `Как выглядит следующий шаг`
9. логику цены
10. `FAQ`
11. финальный CTA

## UX-правила

- состав должен быть виден без долгого скролла;
- блок `что не входит` нельзя прятать;
- CTA должен соответствовать типу решения;
- на mobile нужен sticky CTA;
- related products должны объяснять связь.

## Тон

- предметный;
- честный;
- спокойный;
- дорогой по смыслу;
- без пафоса и “уникальности”.
